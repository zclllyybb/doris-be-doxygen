4// Macro with the boilerplate that makes a type move-only in C++03.
5//
6// USAGE
7//
8// This macro should be used instead of DISALLOW_COPY_AND_ASSIGN to create
9// a "move-only" type. Unlike DISALLOW_COPY_AND_ASSIGN, this macro should be
10// the first line in a class declaration.
11//
12// A class using this macro must call .Pass() (or somehow be an r-value already)
13// before it can be:
14//
15// * Passed as a function argument
16// * Used as the right-hand side of an assignment
17// * Returned from a function
18//
19// Each class will still need to define their own "move constructor" and "move
20// operator=" to make this useful. Here's an example of the macro, the move
21// constructor, and the move operator= from the scoped_ptr class:
22//
23// template <typename T>
24// class scoped_ptr {
25// MOVE_ONLY_TYPE_FOR_CPP_03(scoped_ptr, RValue)
26// public:
27// scoped_ptr(RValue& other) : ptr_(other.release()) { }
28// scoped_ptr& operator=(RValue& other) {
29// swap(other);
30// return *this;
31// }
32// };
33//
34// Note that the constructor must NOT be marked explicit.
35//
36// For consistency, the second parameter to the macro should always be RValue
37// unless you have a strong reason to do otherwise. It is only exposed as a
38// macro parameter so that the move constructor and move operator= don't look
39// like they're using a phantom type.
40//
41//
42// HOW THIS WORKS
43//
44// For a thorough explanation of this technique, see:
45//
46// http://en.wikibooks.org/wiki/More_C%2B%2B_Idioms/Move_Constructor
47//
48// The summary is that we take advantage of 2 properties:
49//
50// 1) non-const references will not bind to r-values.
51// 2) C++ can apply one user-defined conversion when initializing a
52// variable.
53//
54// The first lets us disable the copy constructor and assignment operator
55// by declaring private version of them with a non-const reference parameter.
56//
57// For l-values, direct initialization still fails like in
58// DISALLOW_COPY_AND_ASSIGN because the copy constructor and assignment
59// operators are private.
60//
61// For r-values, the situation is different. The copy constructor and
62// assignment operator are not viable due to (1), so we are trying to call
63// a non-existent constructor and non-existing operator= rather than a private
64// one. Since we have not committed an error quite yet, we can provide an
65// alternate conversion sequence and a constructor. We add
66//
67// * a private struct named "RValue"
68// * a user-defined conversion "operator RValue()"
69// * a "move constructor" and "move operator=" that take the RValue& as
70// their sole parameter.
71//
72// Only r-values will trigger this sequence and execute our "move constructor"
73// or "move operator=." L-values will match the private copy constructor and
74// operator= first giving a "private in this context" error. This combination
75// gives us a move-only type.
76//
77// For signaling a destructive transfer of data from an l-value, we provide a
78// method named Pass() which creates an r-value for the current instance
79// triggering the move constructor or move operator=.
80//
81// Other ways to get r-values is to use the result of an expression like a
82// function call.
83//
84// Here's an example with comments explaining what gets triggered where:
85//
86// class Foo {
87// MOVE_ONLY_TYPE_FOR_CPP_03(Foo, RValue);
88//
89// public:
90// ... API ...
91// Foo(RValue other); // Move constructor.
92// Foo& operator=(RValue rhs); // Move operator=
93// };
94//
95// Foo MakeFoo(); // Function that returns a Foo.
96//
97// Foo f;
98// Foo f_copy(f); // ERROR: Foo(Foo&) is private in this context.
99// Foo f_assign;
100// f_assign = f; // ERROR: operator=(Foo&) is private in this context.
101//
102//
103// Foo f(MakeFoo()); // R-value so alternate conversion executed.
104// Foo f_copy(std::move(f)); // R-value so alternate conversion executed.
105// f = std::move(f_copy); // R-value so alternate conversion executed.
106//
107//
108// IMPLEMENTATION SUBTLETIES WITH RValue
109//
110// The RValue struct is just a container for a pointer back to the original
111// object. It should only ever be created as a temporary, and no external
112// class should ever declare it or use it in a parameter.
113//
114// It is tempting to want to use the RValue type in function parameters, but
115// excluding the limited usage here for the move constructor and move
116// operator=, doing so would mean that the function could take both r-values
117// and l-values equally which is unexpected. See COMPARED To Boost.Move for
118// more details.
119//
120// An alternate, and incorrect, implementation of the RValue class used by
121// Boost.Move makes RValue a fieldless child of the move-only type. RValue&
122// is then used in place of RValue in the various operators. The RValue& is
123// "created" by doing *reinterpret_cast<RValue*>(this). This has the appeal
124// of never creating a temporary RValue struct even with optimizations
125// disabled. Also, by virtue of inheritance you can treat the RValue
126// reference as if it were the move-only type itself. Unfortunately,
127// using the result of this reinterpret_cast<> is actually undefined behavior
128// due to C++98 5.2.10.7. In certain compilers (e.g., NaCl) the optimizer
129// will generate non-working code.
130//
131// In optimized builds, both implementations generate the same assembly so we
132// choose the one that adheres to the standard.
133//
134//
135// WHY HAVE typedef void MoveOnlyTypeForCPP03
136//
137// Callback<>/Bind() needs to understand movable-but-not-copyable semantics
138// to call .Pass() appropriately when it is expected to transfer the value.
139// The cryptic typedef MoveOnlyTypeForCPP03 is added to make this check
140// easy and automatic in helper templates for Callback<>/Bind().
141// See IsMoveOnlyType template and its usage in base/callback_internal.h
142// for more details.
143//
144//
145// COMPARED TO C++11
146//
147// In C++11, you would implement this functionality using an r-value reference
148// and our .Pass() method would be replaced with a call to std::move().
149//
150// This emulation also has a deficiency where it uses up the single
151// user-defined conversion allowed by C++ during initialization. This can
152// cause problems in some API edge cases. For instance, in scoped_ptr, it is
153// impossible to make a function "void Foo(scoped_ptr<Parent> p)" accept a
154// value of type scoped_ptr<Child> even if you add a constructor to
155// scoped_ptr<> that would make it look like it should work. C++11 does not
156// have this deficiency.
157//
158//
159// COMPARED TO Boost.Move
160//
161// Our implementation similar to Boost.Move, but we keep the RValue struct
162// private to the move-only type, and we don't use the reinterpret_cast<> hack.
163//
164// In Boost.Move, RValue is the boost::rv<> template. This type can be used
165// when writing APIs like:
166//
167// void MyFunc(boost::rv<Foo>& f)
168//
169// that can take advantage of rv<> to avoid extra copies of a type. However you
170// would still be able to call this version of MyFunc with an l-value:
171//
172// Foo f;
173// MyFunc(f); // Uh oh, we probably just destroyed |f| w/o calling Pass().
174//
175// unless someone is very careful to also declare a parallel override like:
176//
177// void MyFunc(const Foo& f)
178//
179// that would catch the l-values first. This was declared unsafe in C++11 and
180// a C++11 compiler will explicitly fail MyFunc(f). Unfortunately, we cannot
181// ensure this in C++03.
182//
183// Since we have no need for writing such APIs yet, our implementation keeps
184// RValue private and uses a .Pass() method to do the conversion instead of
185// trying to write a version of "std::move()." Writing an API like std::move()
186// would require the RValue struct to be public.
187//
188//
189// CAVEATS
190//
191// If you include a move-only type as a field inside a class that does not
192// explicitly declare a copy constructor, the containing class's implicit
193// copy constructor will change from Containing(const Containing&) to
194// Containing(Containing&). This can cause some unexpected errors.
195//
196// http://llvm.org/bugs/show_bug.cgi?id=11528
197//
198// The workaround is to explicitly declare your copy constructor.
199//
200
201#pragma once
202
203#define MOVE_ONLY_TYPE_FOR_CPP_03(type, rvalue_type) \
204private: \
205 struct rvalue_type { \
206 explicit rvalue_type(type* object) : object(object) {} \
207 type* object; \
208 }; \
209 type(type&); \
210 void operator=(type&); \
211 \
212public: \
213 operator rvalue_type() { return rvalue_type(this); } \
214 type Pass() { return type(rvalue_type(this)); } \
215 typedef void MoveOnlyTypeForCPP03; \
216 \
217private:
